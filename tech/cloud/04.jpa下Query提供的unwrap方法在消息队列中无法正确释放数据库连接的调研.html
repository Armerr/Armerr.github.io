一、问题产生的原因 ​

同步产品的消息队列消费消息的速度在上线了某个功能后发生了严重的阻塞，通过打印接口执行速度的日志来看，直接调用接口时同步的速度会很快，而同样的产品却在消息队列中的执行要慢非常多甚至阻塞，因此回退了新上的代码后执行变得正常起来，尝试在测试环境中复现问题查明原因。

二、问题定位 ​

一开始怀疑是sql语句问题，但很快被排除了。然后，开始怀疑是数据库连接问题，因为通过debug，可以看到线程阻塞在druid数据库连接池(DruidDataSource)的等待获取资源这个阶段。这很像是之前连接一直没有被释放，只有到了连接超时被强制回收时，后续的消费者才能获取到数据库连接。

问题抽象之后发现确实能够复现问题。对比正常代码与异常代码时发现，两者用了不同的方法返回查询结果。来看这段模拟代码：

java
  public class TestRepository {

    @PersistenceContext
    private EntityManager entityManager;

    public void testSqlQuery() {
        String sql = "select 1 ";
        SQLQuery sqlQuery = entityManager.createNativeQuery(sql).unwrap(SQLQuery.class);
        sqlQuery.list();
    }

    public void testSql() {
        String sql = "select 1 ";
        Query query = entityManager.createNativeQuery(sql);
        query.getResultList();
    }
}

这两个方法的区别是问题代码用了SQLQuery.list()，正常代码用的是Query.getResultList()。在正常代码中常常通过使用unwrap方法设置Query的返回类型做DTO类型转换。

java
sqlQuery.setResultTransformer(Transformers.aliasToBean(Test.class));

从接口访问时，这两个接口都没有问题。但是，通过MQ Listener调用时，testSql方法没有问题，而testSqlQuery会复现同样的问题，数据库连接池资源无法释放，listener线程卡死在等待资源阶段。卡死时，测试环境的消息队列也会阻塞：

这个和预期是不符的，因为当时这么写是因为这段代码的作者看到SQLQuery继承了Query（interface SQLQuery extends Query, SynchronizeableQuery），因此作者认为SQLQuery应该实现了Query的全部功能。

但是再仔细一看，createNativeQuery方法返回的是javax.persistence.Query，而SQLQuery继承的是org.hibernate.Query。这两者其实是不一样的接口。

那么这个差异是导致问题的根本原因吗？仔细查看javax.persistence.Query的实现，发现其实际上封装了一个org.hibernate.Query对象。在调用getResultList()时，实际执行的就是org.hibernate.Query.list()。所以这两者虽然有差异，但实际上这两者的差异并不是导致问题产生的关键。

这时候，在debug调试中，可以发现，在rabbit listener的线程中，Query对象是通过一个代理访问的，这个代理的类型是DeferredQueryInvocationHandler。而正是在这个代理实现中，发生了数据库连接池的回收：

java
//... 实际调用完方法
finally {
    if (queryTerminationMethods.contains(method.getName())) {
        // Actual execution of the query: close the EntityManager right
        // afterwards, since that was the only reason we kept it open.
        EntityManagerFactoryUtils.closeEntityManager(this.em);
        this.em = null;
    }
}

而通过unwrap方法得到的SQLQuery则没有这个代理。这是关键性的区别。可以看到unwrap方法进行了返回的是底层保存的org.hibernate.Query，也就是在这时丢失了代理，也就无法触发回收。

java
public <T> T unwrap(Class<T> tClass) {
    if ( org.hibernate.Query.class.isAssignableFrom( tClass ) ) {
        return (T) query;
    }
    if ( QueryImpl.class.isAssignableFrom( tClass ) ) {
        return (T) this;
    }
    if ( HibernateQuery.class.isAssignableFrom( tClass ) ) {
        return (T) this;
    }

    throw new PersistenceException(
            String.format(
                    "Unsure how to unwrap %s impl [%s] as requested type [%s]",
                    Query.class.getSimpleName(),
                    this.getClass().getName(),
                    tClass.getName()
            )
    );
}

到这里，为什么会卡住的原因是找到了，可是一个新的问题产生了，为什么接口访问时可以自然回收数据库连接呢？

三、接口调用 vs 监听消息 ​

涉及到的orm框架依赖在spring-boot-starter-data-jpa中定义，通过@EnableJpaRepositories注解完成数据库配置。使用时，在Repo里用了这个注解配置entityManager：

java
@PersistenceContext
private EntityManager entityManager;

首先，EntityManager实际是通过一个代理访问，这个代理的位置在org.springframework.orm.jpa包下的SharedEntityManagerCreator，类名是SharedEntityManagerInvocationHandler。它的代理方法完整如下，这里会根据两种情况具体讲解。

java
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    // Invocation on EntityManager interface coming in...

    if (method.getName().equals("equals")) {
        // Only consider equal when proxies are identical.
        return (proxy == args[0]);
    }
    else if (method.getName().equals("hashCode")) {
        // Use hashCode of EntityManager proxy.
        return hashCode();
    }
    else if (method.getName().equals("toString")) {
        // Deliver toString without touching a target EntityManager.
        return "Shared EntityManager proxy for target factory [" + this.targetFactory + "]";
    }
    else if (method.getName().equals("getEntityManagerFactory")) {
        // JPA 2.0: return EntityManagerFactory without creating an EntityManager.
        return this.targetFactory;
    }
    else if (method.getName().equals("getCriteriaBuilder") || method.getName().equals("getMetamodel")) {
        // JPA 2.0: return EntityManagerFactory's CriteriaBuilder/Metamodel (avoid creation of EntityManager)
        try {
            return EntityManagerFactory.class.getMethod(method.getName()).invoke(this.targetFactory);
        }
        catch (InvocationTargetException ex) {
            throw ex.getTargetException();
        }
    }
    else if (method.getName().equals("unwrap")) {
        // JPA 2.0: handle unwrap method - could be a proxy match.
        Class<?> targetClass = (Class<?>) args[0];
        if (targetClass != null && targetClass.isInstance(proxy)) {
            return proxy;
        }
    }
    else if (method.getName().equals("isOpen")) {
        // Handle isOpen method: always return true.
        return true;
    }
    else if (method.getName().equals("close")) {
        // Handle close method: suppress, not valid.
        return null;
    }
    else if (method.getName().equals("getTransaction")) {
        throw new IllegalStateException(
                "Not allowed to create transaction on shared EntityManager - " +
                "use Spring transactions or EJB CMT instead");
    }

    // Determine current EntityManager: either the transactional one
    // managed by the factory or a temporary one for the given invocation.
    EntityManager target = EntityManagerFactoryUtils.doGetTransactionalEntityManager(
            this.targetFactory, this.properties, this.synchronizedWithTransaction);

    if (method.getName().equals("getTargetEntityManager")) {
        // Handle EntityManagerProxy interface.
        if (target == null) {
            throw new IllegalStateException("No transactional EntityManager available");
        }
        return target;
    }
    else if (method.getName().equals("unwrap")) {
        Class<?> targetClass = (Class<?>) args[0];
        if (targetClass == null) {
            return (target != null ? target : proxy);
        }
        // We need a transactional target now.
        if (target == null) {
            throw new IllegalStateException("No transactional EntityManager available");
        }
        // Still perform unwrap call on target EntityManager.
    }
    else if (transactionRequiringMethods.contains(method.getName())) {
        // We need a transactional target now, according to the JPA spec.
        // Otherwise, the operation would get accepted but remain unflushed...
        if (target == null || (!TransactionSynchronizationManager.isActualTransactionActive() &&
                !target.getTransaction().isActive())) {
            throw new TransactionRequiredException("No EntityManager with actual transaction available " +
                    "for current thread - cannot reliably process '" + method.getName() + "' call");
        }
    }

    // Regular EntityManager operations.
    boolean isNewEm = false;
    if (target == null) {
        logger.debug("Creating new EntityManager for shared EntityManager invocation");
        target = (!CollectionUtils.isEmpty(this.properties) ?
                this.targetFactory.createEntityManager(this.properties) :
                this.targetFactory.createEntityManager());
        isNewEm = true;
    }

    // Invoke method on current EntityManager.
    try {
        Object result = method.invoke(target, args);
        if (result instanceof Query) {
            Query query = (Query) result;
            if (isNewEm) {
                Class<?>[] ifcs = ClassUtils.getAllInterfacesForClass(query.getClass(), this.proxyClassLoader);
                result = Proxy.newProxyInstance(this.proxyClassLoader, ifcs,
                        new DeferredQueryInvocationHandler(query, target));
                isNewEm = false;
            }
            else {
                EntityManagerFactoryUtils.applyTransactionTimeout(query, this.targetFactory);
            }
        }
        return result;
    }
    catch (InvocationTargetException ex) {
        throw ex.getTargetException();
    }
    finally {
        if (isNewEm) {
            EntityManagerFactoryUtils.closeEntityManager(target);
        }
    }
}

第一种情况，是通过接口来调用sql。

此时，通过debug可以看到，EntityManager target可以直接从EntityManagerFactoryUtils.doGetTransactionalEntityManager这个方法中取出，所以方法中的isNewEm是false。因此，这部分代码返回是一个原生的Query：

java
Object result = method.invoke(target, args);
if (result instanceof Query) {
    Query query = (Query) result;
    if (isNewEm) {
        Class<?>[] ifcs = ClassUtils.getAllInterfacesForClass(query.getClass(), this.proxyClassLoader);
        result = Proxy.newProxyInstance(this.proxyClassLoader, ifcs,
                new DeferredQueryInvocationHandler(query, target));
        isNewEm = false;
    }
    else {
        EntityManagerFactoryUtils.applyTransactionTimeout(query, this.targetFactory);
    }
}
return result;

上面说过，原生的Query没有关闭连接，回收的操作是在代理中实现的，那么为什么这里明明调用的是原生的Query，实际却回收了连接呢？通过debug发现，在Spring MVC的包中，在网络请求的最后，触发了一个后置方法，它的位置在org.springframework:spring-webmvc中的org.springframework.web.servlet.handler.WebRequestHandlerInterceptorAdapter。

java
@Override
public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex)
        throws Exception {
    this.requestInterceptor.afterCompletion(new DispatcherServletWebRequest(request, response), ex);
}

在这步，requestInterceptor的实现进行了关闭entityManager的操作，之后druid数据库连接池可以正常回收连接。

java
@Override
public void afterCompletion(WebRequest request, Exception ex) throws DataAccessException {
    if (!decrementParticipateCount(request)) {
        EntityManagerHolder emHolder = (EntityManagerHolder)
                TransactionSynchronizationManager.unbindResource(getEntityManagerFactory());
        logger.debug("Closing JPA EntityManager in OpenEntityManagerInViewInterceptor");
        EntityManagerFactoryUtils.closeEntityManager(emHolder.getEntityManager());
    }
}

而另一种情况，在消息队列的消费者处理时，EntityManagerFactoryUtils.doGetTransactionalEntityManager这个方法返回是空，因此isNewEm是true，所以走到了生成代理对象的逻辑，由代理对象的方法关闭entityManager。

那为什么这时返回的是空呢？调查EntityManagerFactoryUtils.doGetTransactionalEntityManager，发现内部实现会用到一个TransactionSynchronizationManager获取entityManager，它的内部实现是通过ThreadLocal保存线程绑定的资源。debug发现，在接口调用时，ThreadLocal经由Spring统一绑定了资源，而在消费消息时并没有绑定资源，这就导致了区别。

四、总结 ​

这个问题的核心在于，Query提供的unwrap方法只是提供了一种便利的与底层实现交互的方式，并不代表能直接用底层实现。

解决这个问题只需要规范查询方法，将有隐患的写法改正。
